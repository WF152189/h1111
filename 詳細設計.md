## 1. はじめに

### 1.1 目的
本資料は、認証・認可機能の詳細な処理仕様を定義し、開発担当者が実装可能なレベルの技術仕様を提供することを目的とする。

### 1.2 位置づけ
本書は詳細設計書として「どのように実装するか（How）」を定義する。

| 設計段階 | 本書の位置づけ | 対象読者 |
|---------|----------------|----------|
| 基本設計 | 業務フロー・機能概要（What） | PM・開発リーダー |
| 詳細設計（本書） | 内部処理ロジック・データ構造（How） | 開発担当者 |
| 実装 | ソースコード | 開発担当者 |

### 1.3 記載規約

本書は以下のルールに従って記載する。

| 項目 | ルール |
|------|--------|
| 処理ロジック | 疑似コードまたはフローチャートで記載 |
| ソースコード | 記載しない（実装フェーズで作成） |
| データ構造 | 項目名・型・桁数・必須/任意を明記 |
| 入出力 | 入力データ・処理・出力データの関係を明確化 |
| エラー処理 | エラー条件・エラーメッセージ・復帰処理を記載 |

## 3. 認証フロー詳細設計

### 3.1 フロー1: セッション有効性チェック（アプリ起動時）

#### 3.1.1 処理概要

| 項目 | 内容 |
|------|------|
| 処理目的 | アプリ起動時に既存セッションの有無を確認し、有効であれば再認証なしで業務画面へ遷移 |
| 起点 | ユーザーが業務画面起動 |
| 入力 | LocalStorage内の業務JWT |
| 出力 | 画面遷移（ダッシュボード or ログイン画面） |

#### 3.1.2 処理ロジック

```
処理名: セッション有効性チェック
入口: アプリ起動時

Step 1. LocalStorageから業務JWT取得
  1-1. localStorage.getItem('business_jwt')実行
  1-2. JWT取得結果を判定
      - JWTあり → Step 2へ
      - JWTなし → Step 3へ

Step 2. JWT有効期限チェック
  2-1. JWTのexpクレームを抽出
  2-2. 現在時刻（Unix時間）取得
  2-3. exp > 現在時刻 を判定
      - TRUE（有効） → Step 4へ
      - FALSE（期限切れ） → Step 3へ

Step 3. RT更新処理実行
  3-1. POST /api/auth/refresh 呼び出し
      - Credentials: 'include'（Cookie自動送信）
  3-2. レスポンス判定
      - 200 OK → Step 4へ
      - 401 Unauthorized → Step 5へ

Step 4. 業務初期画面表示
  4-1. 業務初期データ取得API呼び出し
  4-2. ダッシュボード画面へ遷移
  4-3. 処理終了（正常）

Step 5. ログイン画面へ遷移
  5-1. LocalStorageクリア
  5-2. Azure Entra IDログイン画面へリダイレクト
  5-3. 処理終了（正常）

出口: 処理終了
```

#### 3.1.3 判定基準

| 条件 | 処理 |
|------|------|
| JWTが存在し、exp > 現在時刻 | 業務初期データ取得 → ダッシュボードへ |
| JWTが存在しない、またはexp <= 現在時刻 | RT更新エンドポイント呼び出し |
| RT更新成功（200 OK） | 新規JWT保存 → 業務初期データ取得 → ダッシュボードへ |
| RT更新失敗（401） | Entra ID認証へリダイレクト |

#### 3.1.4 RT更新処理（バックエンド連携）

**処理概要:**

| 項目 | 内容 |
|------|------|
| 処理目的 | JWT期限切れ時にRTを使用して新規業務JWTを取得 |
| 起点 | Step 3.1.2のStep 3（JWT存在なし・期限切れ） |
| 入力 | HttpOnly Cookie内のRT |
| 出力 | 新規業務JWT（LocalStorageに保存） |

**処理ロジック:**

```
処理名: RT更新処理
入口: JWT存在なし・期限切れ検知時

Step 1. RT更新API呼び出し
  1-1. POST /api/auth/refresh リクエスト生成
      - Method: POST
      - Credentials: 'include'（Cookie自動送信）
      - Headers: Content-Type: application/json
  1-2. API呼び出し実行
  1-3. レスポンスステータス判定
      - 200 OK → Step 2へ
      - 401 Unauthorized → Step 3へ
      - その他エラー → Step 3へ

Step 2. 新規JWT保存
  2-1. レスポンスからaccessToken抽出
  2-2. localStorage.setItem('business_jwt', accessToken)実行
  2-3. 呼び出し元へ復帰（正常）

Step 3. エラー処理
  3-1. エラーログ出力
  3-2. LocalStorageクリア
  3-3. Entra IDログイン画面へリダイレクト
  3-4. 呼び出し元へ復帰（異常）

出口: 呼び出し元へ復帰
```

**注意事項:**
- フロントエンドはRTの有無を確認**しない**（HttpOnlyのためアクセス不可）
- 単にRT更新エンドポイントを呼び出し、バックエンドの応答に応じて分岐
- バックエンドがRTの有無・有効性を判定

### 3.2 フロー2: Azure Entra ID認証

#### 3.2.1 処理概要

| 項目 | 内容 |
|------|------|
| 処理目的 | 未ログイン状態のユーザーをAzure Entra IDで認証し、Entra JWTを取得 |
| 起点 | フロー1で「無効」かつ「RT更新不可」と判定 |
| 入力 | ユーザー認証情報（ID/PW、MFA） |
| 出力 | Entra JWT（id_token） |

#### 3.2.2 PKCE認証フロー

**Step 1. PKCEパラメータ生成**

```
処理名: PKCEパラメータ生成
入口: ログイン処理開始時

Step 1. code_verifier生成
  1-1. 32バイトのランダムデータ生成（crypto.getRandomValues）
  1-2. Base64Urlエンコード実行
  1-3. code_verifierとして保存（セッションストレージ）

Step 2. code_challenge生成
  2-1. code_verifierをSHA-256でハッシュ化
  2-2. ハッシュ値をBase64Urlエンコード
  2-3. code_challengeとして保存

Step 3. state生成
  3-1. 16バイトのランダムデータ生成
  3-2. Base64Urlエンコード実行
  3-3. stateとして保存（セッションストレージ）

出口: パラメータ生成完了
```

**Step 2. 認証リダイレクト**

```
処理名: Azure AD 認証リダイレクト
入口: PKCEパラメータ生成完了後

Step 1. 認証URL構築
  1-1. Azure AD認可エンドポイント設定
      - URL: https://login.microsoftonline.com/common/oauth2/v2.0/authorize
  1-2. クエリパラメータ設定
      - client_id: {Azure ADアプリケーションID}
      - response_type: code
      - redirect_uri: {リダイレクトURL}
      - scope: openid profile email
      - code_challenge: {Step 2-2で生成}
      - code_challenge_method: S256
      - state: {Step 3-3で生成}

Step 2. リダイレクト実行
  2-1. window.location.hrefに認証URL設定
  2-2. ブラウザがAzure ADログイン画面へ遷移

出口: リダイレクト完了
```

**Step 3. 認可コード受信・トークン交換**

```
処理名: トークン交換
入口: Azure AD認証完了後、リダイレクトURLに認可コードが付与されて戻る

Step 1. コールバックパラメータ取得
  1-1. URLからcodeパラメータ抽出
  1-2. URLからstateパラメータ抽出

Step 2. state検証（CSRF対策）
  2-1. セッションストレージから保存済みstate取得
  2-2. 受信stateと保存済みstateを比較
  2-3. 一致しない場合 → エラー終了

Step 3. トークンエンドポイントへ送信
  3-1. POSTリクエスト構築
      - URL: https://login.microsoftonline.com/common/oauth2/v2.0/token
      - Content-Type: application/x-www-form-urlencoded
      - Body:
        * client_id: {Azure ADアプリケーションID}
        * grant_type: authorization_code
        * code: {Step 1-1で取得}
        * redirect_uri: {リダイレクトURL}
        * code_verifier: {セッションストレージから取得}
  3-2. POSTリクエスト送信

Step 4. トークン受信
  4-1. レスポンスからid_token抽出
  4-2. id_tokenをメモリに保持（LocalStorage禁止）
  4-3. セッションストレージクリア
      - code_verifier削除
      - state削除

出口: Entra JWT取得完了
```

#### 3.2.3 注意事項

| 項目 | 注意事項 |
|------|----------|
| トークン交換 | フロントエンドから直接Azureへ実施（PKCE対応） |
| Entra JWT保存 | メモリ上のみ保持（LocalStorage禁止） |
| PKCEパラメータ | セッションストレージで管理 |
| state検証 | CSRF対策として必須 |

### 3.3 フロー3: 権限管理API認証・認可

#### 3.3.1 処理概要

| 項目 | 内容 |
|------|------|
| 処理目的 | Entra JWTを基にユーザーのロール・権限を取得し、業務JWTを発行 |
| 起点 | Entra JWTの取得完了後 |
| 入力 | Entra JWT（id_token） |
| 出力 | 業務JWT + リフレッシュトークン |

#### 3.3.2 API仕様

| 項目 | 内容 |
|------|------|
| エンドポイント | POST /api/auth/token |
| 認証 | なし（Entra JWTをBearerトークンで送信） |
| リクエストヘッダー | Authorization: Bearer {Entra_ID_JWT} |
| レスポンス | { "accessToken": "...", "expiresIn": 900, "tokenType": "Bearer" } |
| Cookie | refresh_token（HttpOnly, Secure） |
| エラー | 401: Entra JWT無効・ユーザー不存在 |

#### 3.3.3 処理ロジック

```
処理名: 業務JWT発行
入口: Entra JWT受信時

Step 1. Entra JWT抽出
  1-1. AuthorizationヘッダーからBearerトークン抽出
  1-2. トークン不存在チェック
      - 不存在 → 401エラー返却

Step 2. Entra JWT署名検証
  2-1. Azure AD JWKSエンドポイントから公開鍵取得
      - URL: https://login.microsoftonline.com/common/discovery/v2.0/keys
  2-2. JWT署名検証（RS256）
  2-3. 発行者（iss）検証
      - 検証失敗 → 401エラー返却
  2-4. 有効期限（exp）検証
      - 期限切れ → 401エラー返却

Step 3. ユーザー情報抽出
  3-1. JWTからsubject（sub）抽出 → userId
  3-2. JWTからemail抽出 → email

Step 4. 外部権限管理APIから権限情報取得
  4-1. 外部権限管理API呼び出し
      - URL: {権限管理APIベースURL}/api/users/{userId}/permissions
      - Headers: X-API-Key: {APIキー}
  4-2. レスポンス判定
      - ユーザー不存在 → 401エラー返却
      - APIエラー → 500エラー返却
  4-3. 権限情報取得（roles, permissions）

Step 5. 業務JWT発行
  5-1. JWTペイロード構築
      - iss: "auth-management-api"
      - sub: userId
      - email: email
      - iat: 現在時刻（Unix時間）
      - exp: 現在時刻 + 15分
      - roles: 権限情報から取得
      - permissions: 権限情報から取得
  5-2. RSA秘密鍵で署名（RS256）
  5-3. JWT生成

Step 6. リフレッシュトークン発行
  6-1. 旧RT無効化（is_revoked = TRUE）
  6-2. 新規RT生成（64文字ランダム）
  6-3. RT発行日時設定
  6-4. RT有効期限設定（8時間）
  6-5. RT管理DBに保存

Step 7. RTをHttpOnly Cookieで設定
  7-1. Cookie生成
      - Name: refresh_token
      - Value: 新規RT
      - HttpOnly: true
      - Secure: true
      - Path: /
      - Max-Age: 28800（8時間）
      - SameSite: Strict

Step 8. レスポンス返却
  8-1. HTTPステータス: 200 OK
  8-2. Body: { "accessToken": JWT, "expiresIn": 900, "tokenType": "Bearer" }
  8-3. Headers: Set-Cookie（Step 7-1）

出口: 処理完了
```

#### 3.3.4 JWT署名検証ロジック

```
処理名: Entra JWT署名検証
入口: Entra JWT受信時

Step 1. 公開鍵取得
  1-1. Azure AD JWKSエンドポイントへアクセス
  1-2. JWKSから該当する公開鍵取得（kid一致）

Step 2. 署名検証
  2-1. JWTの署名部を復号
  2-2. 署名アルゴリズム確認（RS256）
  2-3. 公開鍵で署名検証
  2-4. 検証結果判定
      - 失敗 → 検証エラー返却

Step 3. クレーム検証
  3-1. 発行者（iss）検証
      - 期待値: "https://login.microsoftonline.com/{tenant-id}/v2.0"
      - 不一致 → 検証エラー返却
  3-2. 有効期限（exp）検証
      - exp < 現在時刻 → 検証エラー返却
      - 時計ずれ許容: 5分
  3-3. 対象者（aud）検証
      - 期待値: {client-id}
      - 不一致 → 検証エラー返却

出口: 検証結果返却
```

## 4. トークン管理詳細設計

### 4.1 トークン仕様

#### 4.1.1 業務JWT仕様

| 項目 | 仕様 |
|------|------|
| トークン形式 | JWT (JSON Web Token) |
| 署名アルゴリズム | RS256 (RSA Signature with SHA-256) |
| 有効期限 | 15分（expクレーム） |
| 発行者（iss） | "auth-management-api" |
| 保存先 | ブラウザ LocalStorage |
| 使用用途 | 業務API認証・認可 |

**JWTペイロード例:**
```json
{
  "iss": "auth-management-api",
  "sub": "user123",
  "email": "user@example.com",
  "iat": 1713225600,
  "exp": 1713226500,
  "roles": ["admin", "operator"],
  "permissions": ["menu:user", "menu:report", "api:export"]
}
```

#### 4.1.2 リフレッシュトークン仕様

| 項目 | 仕様 |
|------|------|
| トークン形式 | Opaque（ランダム文字列） |
| 長さ | 64文字 |
| 有効期限 | 8時間（更新可能） |
| 保存先 | HttpOnly Cookie |
| Cookie属性 | Secure, HttpOnly, SameSite=Strict, Path=/ |
| RTローテーション | 使用ごとに旧RT無効化・新RT発行 |

**RT生成処理:**

```
処理名: リフレッシュトークン発行
入口: RT発行要求時

Step 1. 旧RT無効化
  1-1. ユーザーIDでRT管理DBを検索
  1-2. 該当ユーザーの全RTに対して is_revoked = TRUE に更新

Step 2. 新規RT生成
  2-1. 64文字ランダム文字列生成（SecureRandom）
  2-2. 発行日時設定（現在時刻）
  2-3. 有効期限設定（現在時刻 + 8時間）
  2-4. ユーザーID紐付け

Step 3. RT管理DBに保存
  3-1. refresh_tokensテーブルにINSERT
  3-2. トランザクションコミット

出口: 新規RT返却
```

**RT格納テーブル:**

| 列名 | 型 | 桁数 | 必須 | デフォルト | 説明 |
|------|----|------|------|-----------|------|
| id | UUID | - | YES | gen_random_uuid() | 主キー |
| user_id | VARCHAR | 255 | YES | - | ユーザーID |
| token | VARCHAR | 64 | YES | - | RTトークン値（UNIQUE） |
| issued_at | TIMESTAMP | - | YES | - | 発行日時 |
| expires_at | TIMESTAMP | - | YES | - | 有効期限 |
| is_revoked | BOOLEAN | - | YES | FALSE | 無効化フラグ |
| created_at | TIMESTAMP | - | NO | CURRENT_TIMESTAMP | 作成日時 |
| updated_at | TIMESTAMP | - | NO | CURRENT_TIMESTAMP | 更新日時 |

**インデックス:**
- idx_refresh_tokens_user_id: user_id
- idx_refresh_tokens_token: token
- idx_refresh_tokens_expires_at: expires_at

### 4.2 RT更新フロー詳細

#### 4.2.1 エンドポイント

**エンドポイント:** `POST /api/auth/refresh`

**リクエスト:**
```
Cookie: refresh_token={RT}
Content-Type: application/json
```

**レスポンス（成功）:**
```json
{
  "accessToken": "{新規業務JWT}",
  "expiresIn": 900,
  "tokenType": "Bearer"
}
```
※ 新規RTもHttpOnly Cookieで自動設定

#### 4.2.2 RT更新処理ロジック

```
処理名: RT更新処理
入口: POST /api/auth/refresh リクエスト受信時

Step 1. RT存在確認
  1-1. Cookieからrefresh_token取得
  1-2. 不存在チェック
      - 不存在 → 401エラー返却

Step 2. RT検証（DB照会）
  2-1. refresh_tokensテーブルをtoken値で検索
  2-2. RT不存在チェック
      - 不存在 → 401エラー返却
  2-3. is_revokedフラグチェック
      - TRUE（無効化済み） → 401エラー返却

Step 3. RT有効期限確認
  3-1. expires_atと現在時刻を比較
  3-2. 期限切れ判定（expires_at < 現在時刻）
      - 期限切れ → RT削除 → 401エラー返却

Step 4. 最新権限情報取得
  4-1. userIdで外部権限管理API呼び出し
  4-2. ユーザー不存在チェック
      - 不存在 → 401エラー返却
  4-3. 権限情報取得（roles, permissions）

Step 5. 新規業務JWT発行
  5-1. JWTペイロード構築
      - iss: "auth-management-api"
      - sub: userId
      - exp: 現在時刻 + 15分
      - roles, permissions: Step 4-3で取得
  5-2. RSA秘密鍵で署名（RS256）
  5-3. JWT生成

Step 6. 新規RT発行（RTローテーション）
  6-1. 旧RT無効化（is_revoked = TRUE）
  6-2. 新規RT生成（64文字ランダム）
  6-3. RT管理DBに保存

Step 7. 新規RTをHttpOnly Cookieで設定
  7-1. Cookie生成
      - Name: refresh_token
      - Value: 新規RT
      - HttpOnly: true
      - Secure: true
      - Path: /
      - Max-Age: 28800（8時間）
      - SameSite: Strict

Step 8. レスポンス返却
  8-1. HTTPステータス: 200 OK
  8-2. Body: { "accessToken": JWT, "expiresIn": 900 }
  8-3. Headers: Set-Cookie（Step 7-1）

出口: 処理完了
```

**RTローテーションの重要ポイント:**
1. **旧RTを即時無効化**: is_revoked = TRUE に更新
2. **新規RT発行**: 64文字ランダム文字列
3. **有効期限**: 新規RTは8時間（または基本設計で設定した値）
4. **権限再照会**: 外部権限管理APIから最新権限を取得（権限変更・アカウント無効化反映）

| MaxAge | 3600秒（プリフライトキャッシュ） |
| 対象パス | /api/** |

### 10.2 フロントエンドエラーハンドリング

```typescript
// error-handler.service.ts
@Injectable({ providedIn: 'root' })
export class ErrorHandlerService {
  
  constructor(private router: Router, private notificationService: NotificationService) {}
  
  handleError(error: HttpErrorResponse): void {
    switch (error.status) {
      case 401:
        // 認証エラー
        this.notificationService.showWarning('セッションが期限切れです。再ログインしてください。');
        this.router.navigate(['/login']);
        break;
        
      case 403:
        // 認可エラー
        this.notificationService.showError('権限が不足しています');
        this.router.navigate(['/error/403']);
        break;
        
      case 500:
        // サーバーエラー
        this.notificationService.showError('システムエラーが発生しました');
        break;
        
      default:
        this.notificationService.showError('エラーが発生しました');
    }
  }
}
```
