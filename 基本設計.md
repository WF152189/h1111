## 1. はじめに

### 1.1 目的
本資料は、認証・認可機能の業務フローおよび機能概要を定義し、関係者間の認識統一を図ることを目的とする。

### 1.2 記載方針（処理の粒度）

本書は基本設計書として「何をするか（What）」を定義する。「どのように実現するか（How）」は外部設計書・詳細設計書で定義する。

| 設計段階 | 焦点 | 記載内容 |
|---------|------|----------|
| 基本設計（本書） | 業務フロー・機能概要 | 処理の目的、主要フロー、条件分岐、技術選定 |
| 外部設計 | 外部インターフェース | APIエンドポイント、リクエスト/レスポンス形式 |
| 詳細設計 | 内部処理ロジック | 署名検証アルゴリズム、DBクエリ、クラス設計 |

記載する（基本設計レベル）:
- 処理の目的・起点・終点
- 主要処理フロー（概要～中程度）
- 条件分岐の概要（成功/失敗パターン）
- 外部システム連携の目的
- セキュリティ要件の概要

記載しない（外部設計・詳細設計レベル）:
- JWT署名検証のロジック詳細
- APIパラメータ定義
- DBクエリ・テーブル設計
- 実装コード・ライブラリ選定

### 1.3 対象範囲
- Azure Entra ID を利用したユーザー認証（OpenID Connect）
- JWT トークンによるセッション管理
- 二重ログインチェック（LocalStorage）

- トークン自動更新・ログアウト処理

## 4. 業務フロー

### 4.1 フロー1：セッション有効性チェック

【目的】
ユーザーが「ログイン」を選択した際、既存セッションの有無を確認し、有効であれば再入力なしで即座にログイン状態へと遷移させる。

【起点】 ログイン画面にて「ログインボタン」を押下

【処理概要】

- Step 1：トークン確認（クライアントサイド）
LocalStorageから業務JWTを取得し、有効期限（exp）を確認。

- Step 2：判定と分岐

- [判定] JWTが有効：
ログイン済みとみなし、初期表示用APIを実行。業務画面へ遷移（終了）。

- [判定] JWTが無効・または存在しない：
RT更新エンドポイントへリクエストを送信（CookieのRT有無はバックエンドで判定）。
  - バックエンド応答が成功（RT有効）：
新しい業務JWTを受信しLocalStorageに保存。業務画面へ遷移（終了）。
  - バックエンド応答が失敗（RT無効・期限切れ）：
Azure Entra ID認証へリダイレクト（フロー2へ）。

【終点】 業務画面表示 または Azure Entra IDリダイレクト

---

### 4.2 フロー2: Azure Entra ID認証（OpenID Connect）

【目的】
未ログイン状態のユーザーをAzure Entra IDで認証し、業務JWT発行の根拠となるEntra JWTを取得する。

【起点】 セッション有効性チェック（フロー4.1）で「無効」かつ「RT更新不可」と判定された場合。

【処理概要】

Step 1: 認証リクエスト送信（SSO開始）
ブラウザからAzure Entra IDの認可エンドポイントへリダイレクト。

方式: Authorization Code Flow with PKCE

Scope: openid, profile, email
- SSO挙動：ブラウザにAzureの有効なセッションCookieが存在する場合、Azure側でのID/PW入力による認証は自動的にスキップされる。

Step 2: ユーザー認証
Azure ADログイン画面にて、ID/PW入力および多要素認証（MFA）を完了させる。

Step 3: トークン交換
認証成功後に発行される「認可コード」を、ブラウザからAzure ADへ送信し、Entra JWT（IDトークン等）を受取る。

【終点】 Entra JWTの取得完了。（そのままフロー3: 業務JWT発行へ遷移）

### 4.3 フロー3: 権限管理API認証・認可

【目的】
Entra JWTを基にユーザーのロール・権限を取得し、業務JWTを発行する。

【起点】Entra JWTの取得完了後

【処理概要】
- Step 1: Entra JWTをBearerトークンとして権限管理APIへ送信
- Step 2: JWT検証（正当性確認）
- Step 3: 外部権限管理APIからユーザーの権限情報を照会
- Step 4: 認可成功 → 業務JWT発行・返却 / 認可失敗 → エラーレスポンス返却
- Step 5: 業務JWTをLocalStorageに保存

【終点】業務JWT取得・保存完了 → 業務画面表示

---

### 4.4 フロー4: アプリケーションアクセス（API認可）

【目的】
業務機能実行時、JWT有効性確認とロールベース権限確認を経てから業務処理を実行する。

【起点】ユーザーが業務機能（メニュー）を選択

【処理概要】
- Step 1: LocalStorageから業務JWTを取得し、有効期限（exp）をブラウザ側で確認
  - JWT有効 → Step 2へ
  - JWT期限切れ → 共通サブフロー（RTによる業務JWT更新処理）へ
    - RT有効 → 新規JWT取得後、Step 2へ
    - RT無効 → セッション期限切れ通知 → フロー1（再認証）へ
- Step 2: 業務JWTをBearerトークンとして付与して外部連携GWへAPIを呼出
- Step 3: 外部連携GWが業務アプリへAPI呼出を転送
- Step 4: 業務アプリが処理を実行し、外部連携GW経由でレスポンスを返却
- Step 5: 画面を表示

【終点】業務処理実行またはエラー返却

---

### 4.5 共通サブフロー: RTによる業務JWT更新処理

【目的】JWT期限切れ時にリフレッシュトークンを使って業務JWTをオンデマンドで更新する。バックグラウンドタイマーは使用せず、JWT期限切れ検知時に実行する。

【起点】フロー2（画面遷移時）またはフロー4（API呼出時）でJWT期限切れを検知

【処理概要（RT有効の場合）】
- Step 1: Cookie内のリフレッシュトークンを権限管理APIへ送信し、業務JWT更新を要求
- Step 2: RTを検証し、外部権限管理APIから最新の権限情報を照会
- Step 3: 新規業務JWTを再発行
- Step 4: 新規リフレッシュトークンを発行（RTローテーション：旧RTを即時無効化）
- Step 5: 新規業務JWTをLocalStorageに保存、新規RTをHttpOnly Cookieに保存
- Step 6: 呼び出し元フローへ復帰

【処理概要（RT無効・期限切れの場合）】
- Step 1: LocalStorageの業務JWTをクリア、HttpOnly CookieのRTを削除
- Step 2: セッション期限切れをユーザーへ通知（モーダル表示）
- Step 3: フロー1（再認証）へ遷移

【終点】新規JWT+RT保存完了（RT有効時）またはフロー1へ遷移（RT無効時）

---

### 4.6 フロー6: ログアウト

【目的】ユーザーセッションを安全に終了する。

【起点】ユーザーがログアウトを要求

【処理概要】
- Step 1: 権限管理APIへリフレッシュトークン無効化を要求
- Step 2: LocalStorageから業務JWTを削除、HttpOnly CookieからRTを削除
- Step 3: ログアウト完了画面を表示

【終点】セッション終了・LocalStorage・Cookie上のトークン完全削除完了

## 6. 認証処理

### 6.1 二重ログインチェック

| 項目 | 内容 |
|------|------|
| 確認方法 | LocalStorageに保存されたJWTの有効期限（exp）を確認 |
| 有効判定 | 現在時刻 < exp → 既存セッションで継続 |
| 無効判定 | JWTなし または 現在時刻 >= exp → Azure Entra ID認証へ |
| 目的 | 不要な再認証を防ぎ、ユーザビリティを確保する |

### 6.2 Azure Entra ID 連携

| 項目 | 内容 |
|------|------|
| 目的 | ユーザー認証・Entra JWTトークン取得 |
| 認証方式 | Authorization Code Flow with PKCE |
| プロトコル | OpenID Connect |
| 指定スコープ | openid, profile, email |
| MFA対応 | Azure Entra ID 側の設定に準拠 |

### 6.3 二段階JWT構造

Entra ID JWT（Azure発行）
  → 権限管理APIへ送信
  → 外部権限管理APIから権限情報を取得
  → 業務JWT（AWS内部発行）
  → LocalStorageに保存 → 以降の業務API認証に使用

| JWT種別 | 発行元 | 用途 | 保存先 |
|---------|--------|------|--------|
| Entra ID JWT | Azure Entra ID | ユーザー認証 | メモリ上（一時） |
| 業務JWT | 権限管理API | 業務API認証・認可 | LocalStorage |

### 6.4 トークン更新

| 項目 | 内容 |
|------|------|
| 更新方法 | Opaqueリフレッシュトークン（HttpOnly Cookie保存）を使用 |
| 更新先 | 権限管理API（内部）※ Azure Entra IDには依存しない |
| 更新タイミング | JWT期限切れ検知時（オンデマンド）。バックグラウンドタイマーは使用しない |
| RTローテーション | RT使用のたびに旧RTを即時無効化し新規RTを発行（盗難検知に対応） |
| 更新時の権限照会 | 外部権限管理APIから最新の権限を照会（権限変更・アカウント無効化を即時反映） |
| 更新失敗時 | セッション期限切れ通知 → フロー1へ再認証 |

## 7. 認可処理

### 7.1 権限モデル（RBAC）

ユーザー → ロール → 権限（パーミッション）

### 7.2 認可レベル

| レベル | 対象 | 制御方法 | タイミング |
|--------|------|----------|------------|
| レベル1: コンテナー認可 | ロール管理コンテナー | Entra JWTを検証・業務JWT発行 | ログイン時 |
| レベル2: API認可 | 業務アプリコンテナー | 業務JWTを検証・エンドポイント単位で権限チェック | API呼出時 |

### 7.3 権限管理APIでの認可フロー

1. Entra JWTの検証
2. 外部権限管理APIから権限情報を照会
3. 認可成功 → 業務JWTを発行
   認可失敗 → エラーレスポンスを返却

### 7.4 APIレベルの認可フロー

1. 業務JWTをBearerトークンとしてAPIリクエストに付与
2. 外部連携GWが業務アプリへ転送
3. 業務アプリが処理を実行
4. 結果を外部連携GW経由でブラウザへ返却

## 8. セキュリティ要件

### 8.1 認証要件

| 要件 | 内容 |
|------|------|
| 外部認証基盤 | Azure Entra ID（OpenID Connect） |
| 認証フロー | Authorization Code Flow with PKCE |
| 二重ログイン防止 | LocalStorageのJWT有効期限チェック |
| MFA | Azure Entra ID 側の設定に準拠 |

### 8.2 トークン管理方針

| 項目 | 内容 |
|------|------|
| 業務JWT保存先 | ブラウザ LocalStorage（CSP設定必須） |
| リフレッシュトークン保存先 | HttpOnly Cookie（Secure属性付き、XSS対策） |
| Entra JWT有効期限 | Azure Entra ID 設定に準拠（メモリのみ保持、LocalStorage非保存） |
| 業務JWT有効期限 | 【要設定：例 15分】 |
| リフレッシュトークン有効期限 | 【要設定：例 8時間〜30日】 |
| 更新方式 | オンデマンドRT更新（RTローテーション） |
| RT形式 | Opaque（ランダム文字列＋DBサーバー管理） |

注意: LocalStorageはXSS攻撃によりトークンが盗難されるリスクがある。
必須対策: Content Security Policy（CSP）の設定
リフレッシュトークンはHttpOnly Cookie（Secure属性）で保護することで、XSSリスクを軽減。

### 8.3 通信セキュリティ

| 要件 | 内容 |
|------|------|
| 通信方式 | HTTPS（TLS 1.2以上） |
| JWT検証 | 署名検証を実施（詳細は詳細設計で定義） |
| スコープ制御 | openid, profile, email を明示指定 |

### 8.4 考慮事項

| 項目 | 内容 |
|------|------|
| Azure Entra ID障害時 | 【要検討: 動作方針定義】 |
| DB障害時 | 【要検討: 認証認可処理の代替方針】 |
| 監査ログ | 認証・認可のアクセスログを記録 |

## 9. エラー処理

### 9.1 エラー分類

| エラー種別 | HTTPステータス | 発生条件 | 処理内容 |
|------------|----------------|----------|----------|
| 認証エラー | 401 | JWT無効・期限切れ | 再認証画面へ遷移 |
| 認可エラー | 403 | 権限なし | エラーメッセージ表示 |
| システムエラー | 500 | 内部エラー | エラー画面表示・ログ記録 |

### 9.2 エラー処理フロー

認証エラー（401）:
  → LocalStorageのJWTをクリア
  → Azure Entra ID 再認証へ遷移
  → 再ログイン完了後、元の操作へ復帰

認可エラー（403）:
  → エラーメッセージをユーザーへ表示
  → 監査ログへ記録
  → 元の画面へ戻る

## 10. 非機能要件

### 10.1 パフォーマンス要件

| 項目 | 目標値 |
|------|--------|
| 初回認証処理時間（Azure AD含む） | 【要設定】秒以内 |
